一、悲观锁的由来：从 “超卖事故” 说起

咱们先从两个场景切入，看懂为啥需要悲观锁：

1. 生活场景：卫生间的 “并发冲突”

你去公司卫生间，推门发现没人但没锁门 —— 这时候如果突然冲进来另一个人，就会发生 “资源冲突”。所以聪明的做法是 先反锁再使用：默认 “一定会有人抢”，提前锁死资源，自己用完再释放。

2. 编程场景：不加锁的 “库存灾难”

这就是 并发冲突：多个线程同时操作共享资源（库存），导致数据混乱。悲观锁的诞生就是为了解决这个问题：操作前先上锁，同一时间只允许一个线程动手，像卫生间反锁一样阻断冲突。

二、悲观锁的核心概念：3 个特点讲透

悲观锁是一种 “先防御后操作” 的并发策略，它默认 “线程一定会抢资源”，所以在访问共享数据前主动获取锁，确保只有持锁线程能操作，其他线程必须排队等待。

关键特点用 “卫生间模型” 类比：

特点 技术解释 生活对应场景
独占性 锁被一个线程持有后，其他线程无权使用 卫生间反锁后，只有里面的人能使用
阻塞性 没拿到锁的线程暂停执行，不会占用 CPU 没抢到卫生间的人在门口排队等待
强一致性 能 100% 避免冲突，但会牺牲性能 虽然排队费时间，但绝对不会撞车

三、Java 中悲观锁的实现方式：2 种锁工具深度解析

Java 里悲观锁分 “隐式锁”（synchronized）和 “显式锁”（ReentrantLock），两者原理和用法差异很大，重点用图示拆解难点。

1. 隐式锁：synchronized（懒人专属，JVM 帮你管锁）

synchronized 是 Java 自带的 “傻瓜式锁”，不用手动开关锁，JVM 自动处理一切。但底层的 Monitor（监视器锁）机制 是难点，必须看图理解。

（1）Monitor 机制可视化解析（核心难点！）

Monitor 本质是个 “管程模型”，像个带门卫的房间，共享资源（比如库存变量）放在房间里，只有拿到钥匙的线程能进。

关键逻辑（对应 synchronized 执行流程）：

  1. 线程进入 synchronized 代码块时，会向 Monitor 申请 “钥匙”（加锁）；

  2. 如果 Monitor 没人持有（owner 为空），线程直接成为 owner，独占资源；

  3. 如果已有线程持有钥匙（比如线程 A），新线程（线程 C）会进入 entrySet 队列 阻塞等待；

  4. 线程执行完代码块，或发生异常，JVM 会自动释放钥匙（owner 置空），并唤醒 entrySet 里的一个线程抢锁；

  5. 如果线程在锁内调用 wait() 方法（比如等待其他线程通知），会暂时释放钥匙，进入 waitSet 队列，被 notify() 唤醒后再回 entrySet 排队。

2. 显式锁：ReentrantLock（灵活王者，手动控锁）

Java 1.5 新增的 java.util.concurrent.locks.Lock 接口实现类，需要手动 lock() 加锁、unlock() 释放锁，功能比 synchronized 强 10 倍。核心难点是 AQS 框架原理。

（1）AQS 框架可视化解析（必懂！）

ReentrantLock 基于 AQS（AbstractQueuedSynchronizer） 实现，本质是个 “带优先级的排队系统”，用双向链表存储等待线程，通过 CAS 操作（Compare And Swap）控制锁的获取释放。

关键逻辑：

  1. 线程调用 lock() 时，AQS 先检查 state 变量（0 = 无锁，>0 = 有锁）；

  2. 如果 state=0，用 CAS 操作把 state 改成 1，线程成为持锁线程；

  3. 如果 state≠0（锁被占用），线程封装成 Node 节点 加入等待队列，进入阻塞状态；

  4. 持锁线程调用 unlock() 时，把 state 改回 0，唤醒队列头节点的线程重新抢锁。

七、总结：悲观锁的 “用武之地” 与 “避坑指南”

1. 什么时候用？

  • ✅ 冲突多（秒杀、抢购）；

  • ✅ 一致性要求高（金融、支付）；

  • ✅ 操作时间长（批量处理、长事务）。

2. 选 synchronized 还是 ReentrantLock？

  • 简单场景（单方法同步）→ 用 synchronized（省心，不易错）；

  • 复杂场景（超时、中断、公平锁）→ 用 ReentrantLock（灵活，功能强）。

3. 核心避坑口诀：

“锁粒度要小，顺序要统一，超时加中断，释放要牢记”。
